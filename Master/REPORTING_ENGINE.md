## 1. Architecture Overview

```
public.records (JSONB)
       │
       ▼
  ETL Sync (warehouse/etl/sync.js)
       │
       ▼
 rpt.fact_fir / rpt.fact_arrest / rpt.fact_pcr / rpt.fact_missing / rpt.fact_uidb
       │
       ▼
  Public report views  (rpt_01_manual_fir, rpt_07_arrested_east_district, ...)
       │
       ▼
  generateParallelReport()  ←  reports.parallel.service.js
       │
       ▼
  ExcelJS fills Master template  (Master/Daily_Diary_ProperHeaders template.xlsx)
       │
       ▼
  HTTP response → browser download  (.xlsx)
```

**Key design decisions:**
- All data flows through the warehouse (`rpt.*` tables), never directly from `public.records` in the report layer.
- The report is generated by writing data INTO a pre-formatted Excel template (not building from scratch). This preserves merged cells, borders, official header formatting, and police department branding.
- Each report view encapsulates its own filtering logic (crime head categories, section keywords, etc.) so the generator code stays generic.

## 2. Data Flow

### Step 1 — Record Creation

A police officer (HC role) creates a record via the frontend. The record is stored in `public.records.data` as a JSONB blob.

```
POST /api/records   →   public.records (JSONB)
```

### Step 2 — Workflow

The record moves through the status chain:

```
DRAFT → PENDING_SHO → DISTRICT_REVIEW → COMPILED → HQ_RECEIVED
```

### Step 3 — ETL Sync

The warehouse ETL (`backend/src/modules/warehouse/etl/sync.js`) reads `public.records` and populates the typed warehouse tables:

| Warehouse Table    | Source Record Type | Key Fields Extracted                          |
|--------------------|--------------------|-----------------------------------------------|
| `rpt.fact_fir`     | `CASE`             | fir_no, sections, crime_head, complainant, IO |
| `rpt.fact_arrest`  | `ARREST`           | arrested_name, sections, crime_head, custody  |
| `rpt.fact_pcr`     | `PCR_CALL`         | gd_no, gd_time, call_head, call_gist          |
| `rpt.fact_missing` | `MISSING`          | name, age, description fields                 |
| `rpt.fact_uidb`    | `UIDB`             | description, found_place, found_date          |

The ETL sync can be triggered via:
```
POST /api/warehouse/sync
```

### Step 4 — Report Generation

The District Officer clicks **"Compile Logs"** in the frontend. This calls:

```
GET /api/daily-diary/export?date=YYYY-MM-DD&psId=...
```

The controller runs `generateParallelReport()` which queries the report views and writes data into the Excel template.

## 3. Warehouse Layer (rpt schema)

### Dimension Tables

| Table                | Purpose                                              |
|----------------------|------------------------------------------------------|
| `rpt.dim_police_station` | Integer PS SK ↔ string PS name mapping           |
| `rpt.dim_district`       | Integer district SK ↔ district name              |
| `rpt.dim_crime_head`     | Crime head SK, normalized code, raw value        |
| `rpt.dim_officer`        | Officer SK, name, PIS                            |
| `rpt.dim_case_status`    | Case status dimension                            |
| `rpt.dim_act_law`        | Act/law dimension                                |

### Fact Tables

All fact tables share a common set of columns:

| Column              | Type      | Description                                      |
|---------------------|-----------|--------------------------------------------------|
| `sk`                | integer   | Surrogate key (auto-generated)                   |
| `source_record_id`  | text      | Corresponds to `public.records.id`               |
| `ps_id`             | integer   | Integer FK to `rpt.dim_police_station`           |
| `district_id`       | integer   | Integer FK to `rpt.dim_district`                 |
| `record_date`       | date      | The diary date for the record                    |
| `workflow_status`   | text      | Current workflow status (DISTRICT_REVIEW, etc.)  |

### Reference Views

| View                  | Purpose                                              |
|-----------------------|------------------------------------------------------|
| `ref_police_station`  | `ps_id` (int) + `ps_name` + `district_id`           |
| `ref_district`        | `district_id` (int) + `district_name`               |
| `ref_crime_head`      | `crime_head_id` + `crime_head_name`                 |
| `ref_case_reg_type`   | `case_reg_type_id` + `code` (MANUAL_FIR / E_FIR)   |

## 5. Parallel Report Generator

**File:** `backend/src/modules/reports/reports.parallel.service.js`

### Entry Point

```js
export const generateParallelReport = async (jobId, filters, filePath, tableNames = null)
```

| Parameter    | Type     | Description                                                    |
|--------------|----------|----------------------------------------------------------------|
| `jobId`      | string   | UUID for logging. Not used for file naming.                    |
| `filters`    | object   | `{ date, psId, districtId }` — scope for the report           |
| `filePath`   | string   | Absolute path where the output `.xlsx` will be written         |
| `tableNames` | string[] | Optional list of `excel_N*` table names to restrict sheets. `null` = all sheets |

### Internal Helpers

#### `resolveIntFilters(filters)`

Converts string IDs to integer warehouse keys:

```
filters.psId = 'PS_NDD_PARLIAMENTSTREET'
  → hierarchy_nodes.where('id', psId)
  → ref_police_station.where('ps_name', node.name_en)
  → returns { psIds: [38], districtId: null }
```

Supports:
- Integer strings (passed through directly)
- Hierarchy node string IDs (e.g., `PS_NDD_PARLIAMENTSTREET`, `DIST_NDD`)
- Comma-separated lists for multiple PS filtering

#### `queryViewData(viewName, date, filters)`

Generic query function used by most sheets:

```js
SELECT v.* FROM <viewName> AS v
WHERE v.diary_record_date = '<date>'
  [AND v.ps_id IN (<psIds>)]       -- if psId filter
  [AND v.district_id = <districtId>]  -- if district filter, no ps filter
```

> **Special case:** Sheet 4 (`rpt_07_arrested_east_district`) is ALWAYS called with `filters: {}` — this view already has `WHERE district_name = 'East District'` built in, so applying a PS/district filter from the user's scope would incorrectly exclude East District records.

#### `populateBlockFromView(options)`

Writes view query results into a specific row range of a worksheet. Uses a **write-in-place** strategy:

1. Snapshot the number of template placeholder rows (`endRow - startRow + 1`)
2. If `data.length > templateRows`: insert only the overflow rows with **one** `spliceRows(insertAt, 0, ...extraRows)` call
3. Overwrite template placeholder rows cell-by-cell with data
4. Clear any remaining unused template rows (when `data.length < templateRows`)

> **Never call `spliceRows(row, deleteCount > 0)`** — deleting rows in ExcelJS corrupts merged cell metadata. This engine inserts overflow rows only.

### Sheet Population Strategy

| Sheet Name                      | Strategy                | Source                          |
|---------------------------------|-------------------------|---------------------------------|
| `1.Manual FIR`                  | Custom populator        | `fir_master` JOIN `ref_case_reg_type` |
| `E-burglary,E-House E-theft, mvt` | `populateBlockFromView` × 4 blocks (bottom-to-top) | `rpt_02` to `rpt_05` |
| `3. Arrested All Heads`         | Custom populator        | `arrest_master` grouped by crime head |
| `4.Arrested East District`      | `populateBlockFromView` | `rpt_07` with `filters: {}`    |
| `5.Arrested Kalandara`          | `populateBlockFromView` | `rpt_08`                        |
| `6.Arrested E-FIR Theft`        | `populateBlockFromView` | `rpt_09`                        |
| `7.Arrested E-FIR MV Theft`     | `populateBlockFromView` | `rpt_10`                        |
| `8.Proclaimed Offenders`        | `populateBlockFromView` | `rpt_11`                        |
| `10.Arrested 24 Hrs List`       | `populateBlockFromView` | `rpt_13`                        |
| `13.Missing, uidb, abandon,trace` | `populateBlockFromView` × 4 blocks | `rpt_18`, `rpt_19`, `rpt_20`, `rpt_21` |
| `14.Women, children Missing`    | Custom populator        | `fact_missing` aggregated       |
| `15.Preventive Action`          | Custom populator        | `pcr_kalandra_master` (gd_time >= 21:00) |
| `16.Inquest Registered`         | `populateBlockFromView` × 2 blocks | `rpt_25`, `rpt_26`  |
| `17.Important Cases`            | Custom populator        | `fir_master` + `arrest_master`  |
| `18.FIR Goswara Summary`        | Custom populator        | `fir_master` grouped by district |
| `19.Financial Fraud Arrest`     | Custom populator        | `arrest_master` + `record_links` (CASE_ARREST) |
| `21.NDPS Action`                | Custom populator        | `fir_master` + `arrest_master` for NDPS sections |

### Sheet Exclusion Logic

```js
const GENERATABLE_SHEETS = new Set([...]);   // 17 auto-populated sheets
const ALWAYS_EXCLUDED_SHEETS = new Set([
  '9.Listed Criminals Action',      // manual entry
  '11.PI Dispose manual, prop. mvt',
  '12.Juveniles Conflict Law',
  '20.Patrolling Checking',
  '22.Servant Verification',
  '23.Mobile Recovered PS',
  '24.Mobile Recovered Summary',
  'Sheet1'
]);
```

When `tableNames` is passed, only the sheets mapped by `OLD_TABLE_TO_NEW_SHEET[tableName]` are kept:

```js
const OLD_TABLE_TO_NEW_SHEET = {
  excel_7arrested_east_district: '4.Arrested East District',
  excel_1manual_fir: '1.Manual FIR',

};
```

## 7. API Endpoint

### `GET /api/daily-diary/export`

**Auth:** Bearer JWT required  
**Scope:** Enforced by role — HC/SHO see only their PS; DISTRICT_OFFICER sees only their district; HQ roles see all.

**Query Parameters:**

| Parameter    | Required | Example                        | Description                                          |
|--------------|----------|--------------------------------|------------------------------------------------------|
| `date`       | No       | `2026-06-22`                   | Report date. Defaults to today (server local time).  |
| `psId`       | No       | `PS_NDD_PARLIAMENTSTREET`      | String hierarchy node ID. Multiple: comma-separated. |
| `districtId` | No       | `DIST_NDD`                     | String hierarchy node ID for district.               |
| `tableNames` | No       | `excel_1manual_fir,excel_7...` | Comma-separated list to restrict which sheets appear. Omit for all 17 sheets. |

**Response:**

```
Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet
Content-Disposition: attachment; filename=Daily_Diary_2026-06-22.xlsx
Body: binary XLSX buffer
```

**Error Responses:**

| Status | Condition                          |
|--------|------------------------------------|
| 400    | Invalid date format                |
| 401    | Missing or invalid Bearer token    |
| 403    | Role not bound to required scope   |
| 500    | Report generation failure          |

### CLI Testing

```bash
cd backend
node scripts/generate_parallel_report_cli.js <jobId> <filtersJson> <outFile>

# Example — all sheets, no scope filter:
node scripts/generate_parallel_report_cli.js test-001 '{"date":"2026-06-22"}' ./scratch/test.xlsx

# Example — Parliament Street only:
node scripts/generate_parallel_report_cli.js test-002 '{"date":"2026-06-22","psId":"PS_NDD_PARLIAMENTSTREET"}' ./scratch/ps-test.xlsx
```

## 8. Frontend Integration

**Component:** `frontend/src/pages/district/CompilationUI.jsx`

The "Compile Logs" button triggers a two-step process:

```
Step 1: POST /api/compilations  { period: date }
        → Creates a compilation record in DB (bundles DISTRICT_REVIEW records)
        → Failure here is non-fatal; export continues with date-based fallback

Step 2: GET /api/daily-diary/export?date=...&[psId=...]&[tableNames=...]
        → Raw fetch() — bypasses axios mock interceptor
        → Response read as ArrayBuffer → Blob → programmatic download
        → Filename: Daily_Diary_<date>_All_Stations.xlsx
                or  Daily_Diary_<date>_<PS_CODE>.xlsx  (if single PS selected)
```

**Token:** Read from `localStorage.getItem('access_token')` — set during login via `useAuth.js`.

**Important:** This export uses **raw `fetch()`**, not the axios `api` instance. This means:
- Mock interceptors in `api.js` do NOT apply
- The real backend is always called, regardless of debug mode
- If the backend is unreachable, `response.ok` will be false and an error toast appears — no file downloads

## 10. Adding a New Sheet

### Step 1 — Create the Report View

Write a SQL view in the `public` schema:

```sql
CREATE OR REPLACE VIEW rpt_NN_your_new_sheet AS
SELECT
  r.id AS record_uid,
  r.record_date AS diary_record_date,
  -- ... your columns here ...
  ps.ps_id,
  ps.district_id
FROM rpt.fact_fir f
JOIN rpt.dim_police_station ps ON ps.sk = f.ps_sk
WHERE <your filter>;
```

Run this in psql or add it to a migration file (`backend/migrations/`).

### Step 2 — Add the Excel Template Sheet

Open `Master/Daily_Diary_ProperHeaders template.xlsx` and add a new worksheet. Make sure the sheet name exactly matches what you'll use in code.

### Step 3 — Add to `GENERATABLE_SHEETS`

In `reports.parallel.service.js`:

```js
const GENERATABLE_SHEETS = new Set([
  // ... existing sheets ...
  'NN.Your New Sheet Name',   // must match the template worksheet name exactly
]);
```

### Step 4 — Add to `OLD_TABLE_TO_NEW_SHEET`

```js
const OLD_TABLE_TO_NEW_SHEET = {
  // ... existing mappings ...
  excel_NN_your_new_sheet: 'NN.Your New Sheet Name',
};
```

### Step 5 — Add the Population Block

Inside `generateParallelReport`:

```js
if (sheetsToKeep.has('NN.Your New Sheet Name')) {
  await populateBlockFromView({
    ws: workbook.getWorksheet('NN.Your New Sheet Name'),
    viewName: 'rpt_NN_your_new_sheet',
    date,
    filters,
    startRow: 4,      // row where data starts (after title + 2 header rows)
    endRow: null,     // null = fill to end of sheet
    colCount: 10,     // number of columns in this sheet
    colMapping: r => [
      r['COLUMN 1'], r['COLUMN 2'], /* ... */
    ]
  });
}
```

### Step 6 — Add to Frontend REPORTS List (optional)

In `frontend/src/pages/district/CompilationUI.jsx`, add a new entry to the `REPORTS` array if you want it to appear in the report selector dropdown:

```js
{ tableName: "excel_NN_your_new_sheet", label: "Your New Sheet", type: "list", num: NN }
```

## 11. Troubleshooting

### Downloaded file is not valid Excel

**Cause:** The backend server is running old code (before the parallel service was introduced). The old code calls `exportDailyDiaryExcel()` in `daily-diary.service.js` which builds from scratch without the proper template.

**Fix:** Restart the backend server.

```bash
# Stop the current server (Ctrl+C), then:
cd backend
npm run dev
```

After restart, verify with this log line appearing when you click "Compile Logs":
```
[ParallelReportService] Compiling jobId: <uuid> to file: <path>
```

### All sheets are empty

**Check 1 — Is the data in the warehouse?**
```bash
cd backend
node --input-type=module << 'EOF'
import db from './src/config/db.js';
const r = await db.raw("SELECT COUNT(*) FROM rpt.fact_fir WHERE record_date='2026-06-22'");
console.log('FIRs:', r.rows[0].count);
await db.destroy();
EOF
```

If 0 → run the seed script: `node scripts/seed-report-test-data.js`

**Check 2 — Did the ETL sync run?**
```
POST /api/warehouse/sync
```

**Check 3 — Do the views return data?**
```bash
node --input-type=module << 'EOF'
import db from './src/config/db.js';
const r = await db.raw("SELECT COUNT(*) FROM rpt_01_manual_fir WHERE diary_record_date='2026-06-22'");
console.log(r.rows[0]);
await db.destroy();
EOF
```

### Sheet 4 (East District) is empty for non-East District users

This is by design. `rpt_07_arrested_east_district` always shows East District data regardless of the requesting user's PS/district scope. The generator passes `filters: {}` for this sheet specifically.

If it IS empty even for the East District user, check that `fact_arrest.district_id = 13` (East District integer ID) for the records in question.

### Wrong date in report title

The title date stamp uses `stampTitleDate()` which replaces the first `DD.MM.YYYY` pattern it finds in cell A1. If A1 doesn't contain a date in that format in the template, the title won't update. Open the template and verify cell A1 on each sheet has a date like `22.06.2026`.

### "No data" on a specific sheet only

1. Identify which `rpt_NN_*` view feeds that sheet (see Sheet Inventory above)
2. Query that view directly:
   ```sql
   SELECT * FROM rpt_NN_view_name WHERE diary_record_date = '2026-06-22' LIMIT 5;
   ```
3. If empty, check the view definition's filter conditions against the seeded data

### CLI vs web server produces different results

The CLI and web server use identical code paths (`generateParallelReport`). Differences are caused by scope filters:

| Path    | `filters` object                                    |
|---------|-----------------------------------------------------|
| CLI     | Only `date` (no scope)                              |
| Web (DISTRICT_OFFICER) | `date` + `districtId = 'DIST_NDD'` → resolves to integer 3 |
| Web (HC/SHO) | `date` + `psId = 'PS_NDD_...'` → resolves to integer PS ID |

To replicate web behavior in CLI:
```bash
node scripts/generate_parallel_report_cli.js test-001 '{"date":"2026-06-22","districtId":"DIST_NDD"}' ./scratch/test.xlsx
```
