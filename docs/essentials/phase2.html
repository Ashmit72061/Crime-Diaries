01 What Phase 1 deliberately left out

Phase 1 was a prototype demonstrating the core record lifecycle. The following were explicitly parked — Phase 2 completes them all. Every item here maps to a concrete task below.

Left out
Legacy data import
Existing registers (Excel / paper scans) are not in the system. Field officers must not re-enter years of records manually.
Left out
JCP / SCP review chain
Workflow goes HC → SHO → District → HQ. The full chain with JCP and SCP levels is not wired up.
Left out
Level Data Contracts
Each level in the hierarchy should see only specific fields. Currently all fields flow to all levels.
Left out
Keycloak / MFA
Auth runs on custom JWT. Production requires Keycloak OIDC with MFA for SHO and above.
Left out
Filter engine
Records list has no advanced filtering. The architecture specifies a full AND/OR filter spec engine with presets.
Left out
Advanced analytics
Only 5 basic count endpoints. Full system: crime heat maps, IO performance, PS vs PS comparison, beat-wise breakdown.
Left out
Proforma report templates
Report engine exists but only 1 generic template. Need 10+ proformas: Arrest Register, PCR Log, Cases Register, etc.
Left out
Admin config UI
Hierarchy, users, field registry are managed by DB script. Need full browser-based admin panel.
Left out
Offline sync (PWA)
App requires internet. PS often have unreliable connectivity. Service Worker + IndexedDB offline-first is required.
Left out
Scheduled reports
Reports are on-demand only. Need auto-generation: daily digest, weekly summary, monthly compilation auto-triggered.
Left out
Send-back target fields
Send-back sends the full form back. Architecture specifies per-field rejection highlighting — only the bad fields light up.
Left out
Filter presets (saved)
No saved search. System should ship with built-in presets: "Today's FIRs", "Pending SHO approval", "Beat 14 this week".
Left out

Audit log exists but SHA256 hash chaining and tamper detection are not implemented.
Left out

No enforcement of submission deadlines. SHOs should be alerted when records sit in queue too long.
Left out
File attachments (MinIO)
File upload fields exist in schema but uploads go nowhere. Need MinIO integration for FIR scans, photos, etc.
Left out
Record duplication check
No protection against accidental duplicate arrest entries. Need FIR number / accused name dedup check on submission.

02 Architecture flexibility — built into every decision

Every Phase 2 feature is designed so future changes never require touching existing code. Adding a new record type, a new hierarchy level, a new report proforma, or a new analytics dimension is always a data operation — not a deployment.

Principle 1
Config over code
Record types, field definitions, hierarchy nodes, report templates, workflow routes — all rows in the DB. Zero code changes for new record types or new PS structures.
Principle 2
Event bus isolation
Every module communicates via RabbitMQ events. A new module (SMS gateway, CCTV integration, beat patrol app) plugs in by subscribing to events — zero changes to existing modules.
Principle 3
JSONB data storage
Record data is JSONB — adding 20 new fields to Arrest records is a Field Registry insert, not an ALTER TABLE. Historical records are untouched. Schema migrations never break old data.
Principle 4
Hierarchy as rows
hierarchy_nodes is a self-referencing tree. Adding a new intermediate level (ACP Range, Division) between District and PS is UPDATE parent_id — no code change, no form changes.
Principle 5
Level data contracts
Each approval level sees only the fields their contract specifies. Change what HQ sees without touching PS forms. New levels inherit sensible defaults.
Principle 6
Database abstraction layer
All DB calls go through a repository pattern. Swapping PostgreSQL for a different DB (or adding Elasticsearch for analytics) is a repository implementation change — no service logic changes.
Principle 7
Workflow as state machine config
Valid transitions are defined in a config object, not hardcoded in if/else chains. Adding a new state (COMMISSIONER_REVIEW) is adding entries to the transition table.
Principle 8
Legacy-compatible data model
All imported legacy records carry is_legacy: true, source_system, and imported_at. They participate in all workflows, filters, reports, and analytics exactly like native records.
The test of a flexible architecture: Can a non-developer admin add a new PS to the hierarchy, define 5 new fields for it, create a custom report template, and set deadlines — all from the browser, with zero backend changes? After Phase 2: yes.

What to test for EVERY endpoint

  • Happy path — valid input, correct role → expected 200/201 response with correct shape
  • Auth guard — no token → 401. Expired token → 401.
  • RBAC guard — wrong role → 403. Example: HC hitting /workflow/queue.
  • Scope guard — valid role but wrong PS/district → 403 or empty set
  • Validation errors — missing required fields → 400 with descriptive message
  • Not found — non-existent ID → 404
  • State violation — e.g. approve a DRAFT (not PENDING_SHO) → 422
  • Pagination — verify page/limit params work, meta.total is accurate
  • Response shape — all fields present, correct types, no extra leakage

Testing priority order (match sprint pace)

  • Sprint A W1: All auth endpoints (login, refresh, logout, /me, change-password, reset). All user CRUD. Hierarchy tree. Role-based access matrix (5 roles × every sensitive endpoint).
  • Sprint A W2: All records CRUD. Full workflow lifecycle (HC→SHO→District→HQ). Send-back with target_fields. Revision history. Audit trail chain.
  • Sprint A W3: Compilation lifecycle. Field registry CRUD. Notifications. Legacy import (when ready).
  • Sprint B W4: Filter engine — all operators, AND/OR nesting, preset CRUD, scope isolation.
  • Sprint B W5: Report generation — all 10+ templates, PDF download, XLSX download, scheduled jobs.
  • Sprint B W6: Analytics — all aggregation endpoints, date range filtering, district scoping.

09 Legacy data import — owned by Dev 2

This is the most complex new feature in Phase 2. Existing paper registers are digitized into Excel/CSV by data entry operators. The import system must accept these, map columns to field registry keys, validate, deduplicate, and load as first-class records with full audit trail.

Critical constraint: Legacy records are immutable once imported. They get status LEGACY_IMPORTED (not DRAFT). They are visible in all analytics, filters, and reports. They cannot be submitted through the workflow (they already happened). But corrections must go through an AMENDMENT sub-workflow.

Import pipeline

// POST /legacy/import — multipart/form-data
// Body: {file: .xlsx/.csv, record_type, ps_id, dry_run: bool}

1. Parse       → ExcelJS reads file → raw row array
2. Map columns → mapping config (UI-defined or default) → {field_key: value}
3. Validate   → required fields, date formats, enum values, phone formats
4. Dedup      → check fir_number OR (accused_name + date_of_arrest) against existing
5. Dry run    → if dry_run=true, return {valid: N, errors: [...], duplicates: [...]} — no write
6. Insert     → bulk INSERT into records with is_legacy=true, status='LEGACY_IMPORTED'
7. Audit      → publish legacy.batch_imported event → Audit Ledger writes provenance row
8. Response   → {imported: N, skipped: N, errors: [{row, reason}]}

Workflow config table (Phase 2 extension)

-- New table: workflow_transitions_config (defines all valid transitions)
CREATE TABLE workflow_transitions_config (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  record_type      VARCHAR NOT NULL,   -- 'ARREST' | 'PCR_CALL' | 'CASE' | '*' (all)
  from_status      VARCHAR NOT NULL,
  to_status        VARCHAR NOT NULL,
  action           VARCHAR NOT NULL,   -- 'SUBMIT' | 'APPROVE' | 'SEND_BACK' | 'SEAL'

  requires_comment BOOLEAN DEFAULT false,

  is_active        BOOLEAN DEFAULT true
);
-- Seed all transitions here. Adding JCP level = INSERT rows. No code change.

SLA / deadline system

-- Cron job runs every 30 minutes
-- Logic: find records where (status_changed_at + sla_hours) < NOW()
SELECT r.id, r.current_status, r.ps_id, wt.sla_hours,
       r.updated_at + (wt.sla_hours || ' hours')::INTERVAL AS deadline
FROM records r
JOIN workflow_transitions_config wt
  ON wt.from_status = r.current_status
WHERE wt.sla_hours IS NOT NULL
  AND r.current_status NOT IN ('ARCHIVED', 'LEGACY_IMPORTED')
  AND r.updated_at + (wt.sla_hours || ' hours')::INTERVAL < NOW() + INTERVAL '2 hours';
-- Publishes: deadline.approaching event → Notifications module

SHA256 hash chain implementation

// On every record_revisions INSERT:
const crypto = require('crypto');
function computeRowHash(revision, prevHash) {
  const content = JSON.stringify({
    record_id:      revision.record_id,
    revision_number: revision.revision_number,
    changed_by:     revision.changed_by,
    changed_at:     revision.changed_at.toISOString(),
    field_changes:  revision.field_changes,
    prev_hash:      prevHash
  });
  return crypto.createHash('sha256').update(content).digest('hex');
}
// Store both prev_hash and row_hash on the row
// POST /admin/audit-verify scans the chain and returns first broken row

New security features

  • MFA — TOTP enforced for SHO, DISTRICT_OFFICER, JCP, SCP, HQ_ADMIN, SYSTEM_ADMIN via Keycloak OTP flow
  • IP allowlist middleware — reject any request not from the intranet CIDR (configured in .env). Nginx handles first; Express has fallback middleware.
  • Session fixation protection — rotate session token on privilege escalation (Keycloak default)
  • CSRF token — double-submit cookie pattern for state-changing requests
  • Per-role rate limits — HC: 200 req/min, SHO: 150 req/min, HQ: 300 req/min, ADMIN: 50 req/min (tighter for admin)
  • Failed login lockout — 5 attempts → 30 min lockout (Keycloak brute force protection)
  • Audit all admin actions — every write to users/hierarchy/fields/contracts publishes admin.config_changed event
  • Tamper detection endpoint — POST /admin/audit-verify scans hash chain, returns {valid, first_broken_at}

21 Future expansion guide — how to extend PHAROS

Design goal achieved: Every item below requires zero changes to existing modules. It is either a DB insert, a new module subscribing to existing events, or a new file in an existing plugin directory.

Adding a new record type (e.g. Beat Patrol Report)

  • Step 1: INSERT INTO field_registry — add all fields for the new type with applicable_record_types = ['BEAT_PATROL']
  • Step 2: INSERT INTO workflow_transitions_config — define the workflow states for the new type (or use record_type = '*' to inherit all existing transitions)
  • Step 3: INSERT INTO report_templates — add report proformas for the new type
  • Step 4: INSERT INTO level_data_contracts — define field visibility at each level
  • Step 5: Frontend — DynamicForm already renders any record type from the registry. Add nav item to the record type picker.
  • Zero changes required to: records CRUD, workflow engine, audit ledger, notifications, filter engine, analytics endpoints (they all accept record_type as a query param), report engine.

Adding a new hierarchy level (e.g. ACP Range between District and PS)

  • Step 1: INSERT INTO hierarchy_nodes — add the ACP Range nodes with node_type = 'ACP_RANGE' and correct parent_id chain
  • Step 2: Keycloak — add ACP_RANGE_OFFICER role
  • Step 3: INSERT INTO workflow_transitions_config — add transitions that include the new level
  • Step 4: INSERT INTO level_data_contracts — define what the ACP level sees
  • Zero changes required to: existing workflows, forms, reports, audit, notifications.

Adding a new external integration (e.g. SMS gateway for senior officers)

  • Step 1: Create src/modules/sms/sms.service.js
  • Step 2: Subscribe to existing events: deadline.approaching, record.status_changed, audit.chain_break_detected
  • Step 3: Send SMS via intranet SMS gateway for the relevant events
  • Zero changes required to: every other module. SMS is just another subscriber on the event bus.

Adding a new analytics dimension (e.g. weather correlation)

  • Step 1: Add the new field (e.g. weather_condition) to field_registry for applicable record types
  • Step 2: Add GET /analytics/by-weather endpoint that aggregates records.data->>'weather_condition'
  • Step 3: Add frontend chart for it
  • Zero changes required to: existing analytics, records, workflow, anything.

Adding a new database backend (e.g. migrate to CockroachDB for HA)

  • Step 1: Implement the Repository interface for CockroachDB (same function signatures, new driver)
  • Step 2: Update db.js connection string
  • Step 3: Migrate data (pgdump-compatible for CockroachDB)
  • Zero changes required to: services, controllers, middleware, frontend.

Adding Elasticsearch for full-text search

  • Step 1: Filter engine already has a routing switch: simple specs → PostgreSQL JSONB, complex specs → Elasticsearch
  • Step 2: Set up Elasticsearch and create index mapping from field_registry
  • Step 3: Subscribe to record.created + record.updated events → index into Elasticsearch
  • Existing queries: continue to work on PostgreSQL. Complex queries automatically route to Elasticsearch once it's up.
