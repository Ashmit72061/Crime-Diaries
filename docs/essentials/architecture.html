PHAROS

Police Hierarchical Automated Reporting & Operations System

12 modules 3 record types 5 hierarchy levels On-premise · intranet-only Offline-capable PWA Hindi/English bilingual ~250 PS · all districts

02 Architecture diagrams

Module architecture — 12 modules in 4 tiers

The amber event bus is the spine — all 12 modules communicate exclusively through it. Cross-cutting tier (purple) runs on every request via middleware and hooks. Core (blue) executes business logic. Features (teal) are independently deployable: each subscribes to events from the bus and knows nothing about other modules. Adding a new feature means a new module subscribing to existing events — nothing existing is touched.

Data flow through hierarchy

District is the critical junction — it simultaneously dispatches compiled data on two routes with different data contracts: the operational chain (JCP → SCP → HQ) and direct HQ reporting. Send-back arrows (amber, dashed) carry the target_fields array — the reviewing level specifies exactly which fields need correction without rejecting the whole record. HQ eventually receives data from both paths.

01 Architectural pillars

Pillar 1
Dynamic field registry
Every field is a row in field_registry, not a DB column. Forms render from this table at runtime. Adding, removing, or modifying a field requires only an INSERT or UPDATE — no migration, no deployment, no code change.
Pillar 2
Immutable event ledger
Every field change at every level is written to an append-only record_revisions table. Nothing is ever deleted. Field-level diffs track who, what, old value, new value, when, from which IP. Hash-chained for tamper detection.
Pillar 3
Level-aware data contracts
What District sends to JCP is not what it sends to HQ. These contracts live in a level_data_contracts config table. Changing what fields flow on which route requires a config edit — not a deployment.
Pillar 4
Event-bus isolation
Modules never call each other directly. All inter-module communication is async via RabbitMQ. Adding a new feature means adding a new module that subscribes to events. Nothing existing can break because nothing existing is coupled to the new module.
Pillar 5
Hierarchy as configuration
PS → District → JCP → SCP → HQ is a single self-referencing hierarchy_nodes table. Restructuring (move a PS, rename a district, add a new level) = row updates. Zero code changes required.

02 Data architecture

Three record types (ARREST, PCR_CALL, CASE) share one physical records table with a JSONB data column. The field registry governs what lives inside that blob. PCR_CALL is independent; CASE and ARREST link via fir_number through a case_arrest_links junction table — they can be queried jointly or independently depending on the use case.

records

-- All three master types use this single table
records (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  record_type     VARCHAR NOT NULL  -- 'ARREST' | 'PCR_CALL' | 'CASE'
  ps_id           UUID NOT NULL REFERENCES hierarchy_nodes(id),
  district_id     UUID NOT NULL REFERENCES hierarchy_nodes(id),
  data            JSONB NOT NULL DEFAULT '{}',  -- ALL field values live here
  current_status  VARCHAR NOT NULL DEFAULT 'DRAFT',
  current_level   VARCHAR NOT NULL DEFAULT 'PS',
  version         INTEGER NOT NULL DEFAULT 1,
  created_by      UUID NOT NULL REFERENCES users(id),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
)

field_registry

-- The schema itself — every field that can exist in any record
field_registry (
  id                       UUID PRIMARY KEY,
  field_key                VARCHAR UNIQUE NOT NULL,  -- "accused_name", "fir_no", "date_of_arrest"
  field_type               VARCHAR NOT NULL,
    -- TEXT | NUMBER | DATE | DATETIME | SELECT | MULTISELECT | FILE | BOOLEAN | PHONE
  applicable_record_types  VARCHAR[] NOT NULL,  -- ['ARREST','CASE'] or ['PCR_CALL'] etc.
  label_en                 VARCHAR NOT NULL,    -- English label shown on form
  label_hi                 VARCHAR NOT NULL,    -- Hindi label shown on form
  options                  JSONB,               -- [{value, label_en, label_hi}] for SELECT fields
  validation_rules         JSONB,               -- {required, min, max, regex, pattern}
  visible_to_levels        VARCHAR[] NOT NULL DEFAULT '{PS,DISTRICT,JCP,SCP,HQ}',
  editable_by_levels       VARCHAR[] NOT NULL DEFAULT '{PS}',
  introduced_at_level      VARCHAR NOT NULL DEFAULT 'PS',
  section                  VARCHAR,             -- form section grouping
  sort_order               INTEGER DEFAULT 0,
  is_active                BOOLEAN NOT NULL DEFAULT TRUE  -- soft-delete; historical data preserved
)

record_revisions

-- APPEND-ONLY. NEVER update or delete rows here.
record_revisions (
  id               UUID PRIMARY KEY,
  record_id        UUID NOT NULL REFERENCES records(id),
  revision_number  INTEGER NOT NULL,
  changed_by       UUID NOT NULL REFERENCES users(id),
  changed_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  level            VARCHAR NOT NULL,
  change_type      VARCHAR NOT NULL,  -- 'CREATE'|'UPDATE'|'STATUS_CHANGE'|'LEVEL_TRANSITION'

  row_hash         VARCHAR            -- SHA256 of this row's content
)

03 Module descriptions

Cross-cutting
IAM
Users, roles, permissions, session management. Backed by Keycloak OIDC. Every API call passes through IAM middleware before any business logic runs. Manages role hierarchy for 1000+ users across all PS.
Cross-cutting
Field registry
Source of truth for all field definitions. Exposes a read API consumed by Data Entry, Filters, Reports, and Analytics. A change here propagates everywhere automatically — no sync needed.
Cross-cutting
Audit ledger
Subscribes to record.changed events from all other modules. Writes append-only rows to record_revisions. Cannot be called directly — only via the event bus — making it impossible to bypass.
Cross-cutting
Admin config
Manages hierarchy_nodes, level_data_contracts, user management, field registry UI. Only SYSTEM_ADMIN can write. District Admins can read relevant portions. All changes are audit-logged.
Core
Data entry
Renders forms dynamically from the Field Registry. Handles draft auto-save (every 30s + on change), file uploads to MinIO, bilingual label switching, per-field validation, and initial records row creation. Publishes record.created and record.updated events.
Core
Workflow
Finite state machine for record lifecycle. Validates every transition against role permissions. On SEND_BACK, requires comment and optionally target_fields. Publishes transition events consumed by Notifications and Audit Ledger. Illegal transitions throw 403 — not silently ignored.
Core
Compilation
District-level and above. Collects all approved records for a period, applies the relevant Level Data Contract to select/transform fields, and produces compilations rows. Handles the two parallel routes (ops chain vs. direct HQ) as separate compilation jobs triggered by the same event.
Core
Sync
Service Worker + IndexedDB. Intercepts API calls when offline, queues as sync_ops, replays in strict order on reconnect. Conflict rule: DRAFT = last-write-wins; ≥ PENDING_SHO = server wins. Notifies user of any discarded local edits.
Feature
Filters
Accepts a JSON filter spec (AND/OR tree). Translates to PostgreSQL JSONB query (simple) or Elasticsearch DSL (complex/full-text). Every field in the registry is automatically filterable. Preset management with SYSTEM / ROLE / USER scoping.
Feature
Reports
Template-based proforma generation. Templates define sections, field mappings, computed aggregates, headers, footers. Renders to PDF via Puppeteer or Excel via ExcelJS. New proforma = new template row in Admin UI. Zero deployments for new report types.
Feature
Analytics
Dashboards scoped per hierarchy level. Comparative reports (day/week/month/year), PS vs PS, District vs District, beat-wise, crime-head-wise, IO performance. Powered by Elasticsearch aggregations. Renders trend charts and heat maps directly in the browser.
Feature
Notifications
Subscribes to record.submitted, record.sent_back, compilation.due, deadline.approaching events. Delivers in-app alerts, maintains per-user pending count in Redis (live badge), and stores notification history. Extensible to SMS/email within intranet without touching other modules.

04 Hierarchy configuration

All relationships are rows in hierarchy_nodes. The Admin Config module provides a UI to add, rename, or restructure nodes without touching code.

HQ (root, parent_id = NULL)
 └── Spl. CP — Zone A
      └── Jt. CP — District Group North
           ├── District: Central Delhi
           │    ├── PS: Connaught Place
           │    ├── PS: Parliament Street
           │    └── PS: Chanakyapuri
           └── District: North Delhi
                ├── PS: Civil Lines
                └── PS: Timarpur
 └── Spl. CP — Zone B
      └── ...
To move a PS to a different District: UPDATE hierarchy_nodes SET parent_id = <new_district_id> WHERE id = <ps_id>. Historical records retain their original ps_id and district_id for audit accuracy. Only future records use the new assignment.

Level Data Contracts define what fields flow on each route. The Compilation module reads these at compile time. If contracts change mid-week, the next compilation uses the new contract; past compilations are unaffected.

05 Workflow state machine

DRAFT └─ [HC submits] ──────────────────────────────────────→ PENDING_SHO │ ┌─ [SHO sends back + comment] ──┘ ↓ │ [SHO approves] SENT_BACK_HC ↓ │ DISTRICT_REVIEW [HC resubmits] │ └───────────────────┤ ├─ [District sends back] ──→ SENT_BACK_PS ──→ PENDING_SHO │ [District compiles] ↓ COMPILED ├─── [ops chain] ──→ JCP_REVIEW ──→ SCP_REVIEW ──┐ └─── [direct HQ] ─────────────────────────────→─┤ HQ_RECEIVED └─ [HQ seals] ──→ ARCHIVED ■
On SEND_BACK: the target_fields array in workflow_transitions specifies exactly which fields need correction. The form highlights only those fields. This prevents rejecting a complete record when only one field is wrong.
All transitions are validated by the Workflow module against role permissions before being applied. Illegal transitions (e.g. HC approving their own record, or District accessing JCP-level records) throw 403 and are logged in the audit ledger.

06 Filter engine

Filters are JSON specs — an AND/OR tree of field-operator-value conditions. The backend parses into PostgreSQL JSONB queries (simple, fast) or Elasticsearch DSL (full-text, complex multi-field). Every field added to the registry is automatically filterable — no code change needed for new fields.

-- Filter spec format: AND/OR condition tree
{
  "logic": "AND",
  "conditions": [
    { "field": "crime_head",  "operator": "IN",      "value": ["Robbery", "Theft"] },
    { "field": "beat_no",     "operator": "EQ",      "value": "14" },
    {
      "logic": "OR",
      "conditions": [
        { "field": "date_of_arrest", "operator": "BETWEEN",   "value": ["2024-01-01", "2024-01-31"] },
        { "field": "status",          "operator": "EQ",         "value": "UNDER_INVESTIGATION" }
      ]
    }
  ]
}
Field typeAvailable operators
TEXTeq, not_eq, contains, starts_with, ends_with, is_empty, is_not_empty
NUMBEReq, not_eq, gt, gte, lt, lte, between
DATE / DATETIMEeq, before, after, between, last_n_days, this_week, this_month, this_year
SELECTin, not_in, is_empty
MULTISELECTcontains_any, contains_all, is_empty
BOOLEANis_true, is_false
FILEhas_attachment, no_attachment

Preset examples

System-level presets (available to all): "Today's FIRs" · "Pending SHO approval" · "Cases under S.302" · "PCR calls — no action taken" · "Arrests this month by crime head" · "Beat 14 this week" · "IO-wise pending cases". 07 Version control

Every save at every level generates a record_revisions row. The UI shows a full diff timeline per record. At ARCHIVED status, the chain is hash-sealed.

-- field_changes payload: every field that changed in this save
[
  {
    "field_key":  "accused_name",
    "label_en":   "Name of arrested person",
    "label_hi":   "गिरफ्तार व्यक्ति का नाम",
    "old_value":  "Ramesh Singh",
    "new_value":  "Rakesh Singh"
  },
  {
    "field_key":  "place_of_arrest",
    "label_en":   "Place of arrest",
    "old_value":  "Lajpat Nagar",
    "new_value":  "Lajpat Nagar Market, New Delhi"
  }
]

Tamper detection: each record_revisions row stores prev_hash (SHA256 of the previous row) and row_hash (SHA256 of this row's content). Any retroactive insert or modification breaks the chain at that point — detectable by a simple sequential hash-verification scan.

The timeline view shows revisions in reverse-chronological order, with bilingual labels, color-coded old/new values, and the actor's name, badge number, level, and timestamp. Any user with read access to the record can view its complete history.

08 Offline sync

Create / edit draftsFinal submission (state machine transition)
View all local PS recordsReceiving send-backs from SHO
View cached filter presetsCross-PS or cross-District data
Search local cached recordsReal-time report generation

Offline operations queue as sync_ops in IndexedDB. On reconnect, the Sync Module replays the queue in strict creation order. Conflict rule: DRAFT records use last-write-wins (local beats server only if server version hasn't advanced). Records at PENDING_SHO or above are server-authoritative; local edits are silently discarded and the user is shown a notification: "Record has advanced — your local changes were not applied."

10 Analytics

Each hierarchy level sees analytics scoped to their access: PS sees its own data; District sees all PS under it; HQ sees everything. All dashboards include a comparative period selector (previous day / week / month / year / custom range).

LevelAvailable analytics
PS / SHODaily record counts · Crime head breakdown · IO-wise pending · Beat-wise heat map
DistrictAll above + PS vs PS comparison · Weekly / monthly trends · Top crime heads by PS · Revision frequency by HC (data quality proxy)
JCP / SCPDistrict vs District · Zone-level summaries · Escalation rates · Compilation timeliness
HQAll-Delhi views · Year-over-year · Cross-district crime type correlation · Section-of-law distribution · Officer performance metrics

Powered by Elasticsearch aggregations — faster than PostgreSQL GROUP BY on JSONB at scale. The Analytics module also reads from compiled compilations rows for district-level and above queries, which are pre-aggregated and much faster than querying raw records.

14 System logging

PHAROS uses Winston as the NestJS application logger, morgan for HTTP access logs, and winston-elasticsearch to ship every log line in real-time to the ELK stack. All logs are structured JSON — machine-parseable and fully indexed in Kibana for search, alerting, and post-mortem analysis.

Winston setup (logger.config.ts)

// npm install winston nest-winston winston-elasticsearch
import { WinstonModule } from 'nest-winston';
import * as winston from 'winston';
import { ElasticsearchTransport } from 'winston-elasticsearch';

export const loggerConfig = WinstonModule.createLogger({
  level: process.env.LOG_LEVEL || 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.errors({ stack: true }),   // include full stack on error objects
    winston.format.json()
  ),
  defaultMeta: { service: 'pharos-api' },
  transports: [
    // Console: colourised in dev, raw JSON in production
    new winston.transports.Console({
      format: process.env.NODE_ENV === 'production'
        ? winston.format.json()
        : winston.format.combine(winston.format.colorize(), winston.format.simple())
    }),
    // File: errors only (50 MB rotation, 5 files max = 250 MB max)
    new winston.transports.File({
      filename: 'logs/error.log', level: 'error',
      maxsize: 50 * 1024 * 1024, maxFiles: 5
    }),
    // File: all levels (100 MB rotation, 10 files = 1 GB max)
    new winston.transports.File({
      filename: 'logs/combined.log',
      maxsize: 100 * 1024 * 1024, maxFiles: 10
    }),
    // Elasticsearch: ships to ELK in real-time, indexed as pharos-logs-YYYY.MM.DD
    new ElasticsearchTransport({
      level: 'info',
      indexPrefix: 'pharos-logs',
      clientOpts: { node: process.env.ELASTICSEARCH_URL }
    })
  ],
  // Catch process-level crashes before Node exits
  exceptionHandlers: [
    new winston.transports.File({ filename: 'logs/exceptions.log' }),
    new ElasticsearchTransport({
      level: 'error', indexPrefix: 'pharos-crashes',
      clientOpts: { node: process.env.ELASTICSEARCH_URL }
    })
  ],
  rejectionHandlers: [
    new winston.transports.File({ filename: 'logs/rejections.log' })
  ]
});

Correlation ID middleware

// correlation.middleware.ts — every request gets a unique ID that threads through all logs
@Injectable()
export class CorrelationMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction) {
    const id = (req.headers['x-correlation-id'] as string) || uuidv4();
    req['correlationId'] = id;
    res.setHeader('X-Correlation-Id', id);  // echo back so client can correlate
    next();
  }
}

// Usage in any service method:
this.logger.log({
  message: 'Record submitted to SHO',

HTTP logging interceptor

// logging.interceptor.ts — logs every request and response (or error) automatically
@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  intercept(ctx: ExecutionContext, next: CallHandler): Observable<any> {
    const req = ctx.switchToHttp().getRequest();
    const start = Date.now();
    return next.handle().pipe(
      tap(() => this.logger.log({
        level: 'http', message: `\${req.method} \${req.url}`,
        correlationId: req.correlationId, userId: req.user?.id,
        duration: Date.now() - start,
        status: ctx.switchToHttp().getResponse().statusCode
      })),
      catchError(err => {
        this.logger.error({
          message: err.message, stack: err.stack,
          correlationId: req.correlationId, userId: req.user?.id,
          path: req.url, method: req.method, duration: Date.now() - start
        });
        throw err;
      })
    );
  }
}

Crash debugging

Winston's exceptionHandlers and rejectionHandlers catch Node.js-level crashes before the process exits. For OOM or SIGKILL (which bypass Winston), PM2 captures the exit code and writes to the system journal, then auto-restarts. Run with NODE_OPTIONS=--enable-source-maps so stack traces in production point to TypeScript source lines, not compiled JS offsets.

For memory leak investigation: start the app with --inspect=0.0.0.0:9229 on the app server (intranet-only) and connect Chrome DevTools remotely. For automated heap snapshots on OOM: add --heapsnapshot-signal=SIGUSR2 to Node options and send the signal from the monitoring server when Grafana's memory alert fires. Snapshots land in the working directory and can be loaded in Chrome Memory tab.

For slow-query root-cause: PostgreSQL is configured with log_min_duration_statement = 500 (logs any query taking >500ms). These logs are shipped to ELK via Filebeat from the PG log directory.

ELK index lifecycle management (ILM) handles rotation and deletion automatically. Kibana dashboards are set up for: error rate over time, slow queries, auth failure frequency, record submission rate by PS, and module error distribution. Grafana alerting fires on error rate spikes and triggers a PagerDuty-equivalent notification to the on-call admin's intranet number.

16 Security architecture

LayerMechanism
Network perimeterIP allowlist to police intranet CIDR only. No external internet access to any service. Nginx as single ingress point.
TransportTLS 1.3 for all HTTP traffic within intranet. mTLS between backend services (NestJS ↔ PostgreSQL, RabbitMQ, Redis, MinIO).
AuthenticationKeycloak OIDC. JWT access tokens (1h expiry) + refresh tokens (7 days, rotated on use). MFA enforced for all roles SHO and above.
AuthorizationServer-side RBAC on every request. Row-level security in PostgreSQL for PS/District data isolation. Scope checks in NestJS guards before business logic.
Data at restAES-256 encryption on all PostgreSQL, Elasticsearch, MinIO, and Redis volumes. Keycloak token store encrypted separately.
Audit trailAppend-only record_revisions with SHA256 hash chaining. IP and user-agent logged on every write. Deletion of any audit row is detectable and blocked at DB level via trigger.
Rate limiting100 requests/minute per authenticated user. 10 login attempts/hour per IP. Both enforced at Nginx gateway.
Backup securityDaily encrypted pg_dump to backup server (AES-256 passphrase stored in Keycloak vault). MinIO snapshots replicated to backup. 30-day retention.
