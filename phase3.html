01 What we are building

Feature 1A
Predefined Proformas
Fixed police register formats (arrest summary, PCR log, etc). Field set is locked — defined in DB. User only picks date range and jurisdiction. New proforma = new DB row, zero code change.
Feature 1B
Custom Report Builder
User picks record types and ticks any fields they want. Multiple record types = multiple sheets in one Excel. Optional: save selection as a named reusable template.
Feature 1C
Daily Diary
One Excel file, 34 sheets, each sheet = one police proforma. District Officer picks date + PS. Each sheet is populated with live DB data. Downloadable from Compile tab.
Feature 2 & 3
Bulk Import
Download blank template (auto-generated from field_registry with Excel dropdowns). Fill it. Upload for dry-run validation. Review errors row by row. Confirm to commit. Same flow for PS-level entry and legacy data.
The one rule everything follows: field_registry is the only place that knows what fields exist. Report column lists, import template headers, validation rules — all read from field_registry at runtime. Adding a field to the registry automatically makes it available everywhere.

02 Complete pipeline

Node.js handles all HTTP. Python handles all file generation. They talk via RabbitMQ. This separation means you can scale the Python worker independently and swap the generation engine without touching any API.

CLIENT (browser) │ POST /api/reports/generate { template_id, filters, format } │ NODE.JS API │ 1. Validate request (template exists, user has scope access) │ 2. INSERT report_jobs { status: 'PENDING' } │ 3. publish('report.requested', { job_id, template_id, filters, format, user_id }) │ 4. Return HTTP 201 { job_id, status: 'PENDING' } ← immediate, no waiting │ RABBITMQ (exchange: 'pharos', topic routing) │ routing key: 'report.requested' │ PYTHON WORKER │ 1. Consume message from queue 'report-generation-queue' │ 2. Load report_jobs row by job_id (query same PostgreSQL) │ 3. Load template_definition from report_templates (or custom_definition from job) │ 4. Load field metadata from field_registry for fixed_fields[] │ 5. Build pandas query against records table │ → SELECT records.data->>'field_key' for each field │ → WHERE record_type + data_filter + date range + jurisdiction │ 6. Generate file: │ → EXCEL: openpyxl workbook (one sheet per type, or 34 sheets for daily diary) │ → PDF: WeasyPrint HTML → PDF │ → CSV: pandas df.to_csv() │ 7. Write file to REPORTS_DIR (shared volume) │ 8. UPDATE report_jobs SET status='READY', file_path=... │ 9. publish('report.generated', { job_id, file_size }) │ NODE.JS API (client polling) │ GET /api/reports/status/:jobId → reads report_jobs.status │ When READY: client calls GET /api/reports/download/:jobId │ Node.js streams file from disk to browser │ DONE File downloaded in browser
Shared volume: Both Node.js and Python write/read from the same REPORTS_DIR path (e.g. ./generated-reports). In Docker this is a named volume mounted in both containers. Python writes the file, Node.js serves it. Neither needs to know how the other works.

04 template_definition — how it works

This is the JSONB column on report_templates that replaces every hardcoded field list in the current controller. The Python worker reads this at generation time to know what to query and how to render it.

Structure for PROFORMA type

{
  "filter_spec": {                        ← PART 1: what records to pull
    "record_type": "CASE",
    "data_filter": { "crime_head": "E_BURGLARY" }
    -- goes into: WHERE records.data @> '{"crime_head":"E_BURGLARY"}'::jsonb
    -- user's date/PS filters stack ON TOP of this
  },

  "fixed_fields": [                        ← PART 2: which columns, in order
    "ps_name", "e_fir_no", "us_section",
    "complainant_name", "father_husband_name"
    -- these are field_keys from field_registry
    -- label_en (column header) is fetched from field_registry at runtime
    -- changing a label in field_registry → report header updates automatically
  ],

  "computed_fields": [                     ← PART 3: derived columns (not from DB)
    { "label_en": "Sr. No.", "formula": "ROW_INDEX" }
    -- ROW_INDEX: sequential 1,2,3... added during render
    -- placed BEFORE fixed_fields columns in output
  ],

  "header": {                              ← controls top rows of Excel sheet
    "title_en": "Details of E-Burglary Cases Registered in PS",
    "title_hi": "पुलिस थाने में पंजीकृत ई-सेंधमारी के मामले",
    "short_name_en": "2.E-Burglary Cases",  ← Excel sheet tab name (max 31 chars)
    "show_date_range": true,
    "show_ps_name": true,
    "layout": "A4_LANDSCAPE"
  }
}

Structure for COMPOSITE type (Daily Diary)

{
  "sub_template_ids": [          ← ordered list of PROFORMA template UUIDs
    "<uuid-1-manual-fir>",
    "<uuid-2-e-burglary>",
    "<uuid-3-e-house-theft>",
    -- ... all 34 proforma IDs in order
  ],
  "user_selectable": true        ← user can deselect individual sheets
  -- Python worker loops through this list
  -- each ID → one worksheet in the Excel workbook
  -- worksheet name = that template's header.short_name_en
}

How the Python worker uses it

# python_worker/generator.py

def generate_report(job_id):
    job = db.execute("SELECT * FROM report_jobs WHERE id = %s", [job_id]).fetchone()

    # Step 1 — resolve the field spec
    if job.template_id:
        template = db.execute("SELECT * FROM report_templates WHERE id = %s", [job.template_id]).fetchone()
        definition = template.template_definition   # already a dict (psycopg2 parses JSONB)
    else:
        definition = job.custom_definition          # came from request body, stored in job

    # Step 2 — composite (daily diary) branches here
    if template.template_type == 'COMPOSITE':
        wb = generate_composite(definition, job.filters)
    else:
        wb = generate_single_sheet(definition, job.filters)

    # Step 3 — write file
    wb.save(job.file_path)
    db.execute("UPDATE report_jobs SET status='READY' WHERE id = %s", [job_id])
    publish_event('report.generated', { 'job_id': job_id })

def generate_single_sheet(definition, user_filters):
    # Load field metadata from field_registry
    field_keys = definition['fixed_fields']
    fields_meta = load_field_registry(field_keys)   # { field_key: { label_en, field_type } }

    # Build SQL — extract each field_key from JSONB data column
    selects = [f"records.data->>'{k}' as {k}" for k in field_keys]
    sql = f"""
        SELECT {', '.join(selects)}, hn.name_en as ps_name, records.record_date
        FROM records
        LEFT JOIN hierarchy_nodes hn ON records.ps_id = hn.id
        WHERE records.record_type = %(record_type)s
        AND records.data @> %(data_filter)s::jsonb
        AND records.record_date BETWEEN %(date_from)s AND %(date_to)s
    """
    if user_filters.get('ps_id'):
        sql += " AND records.ps_id = %(ps_id)s"

    df = pd.read_sql(sql, engine, params={
        'record_type': definition['filter_spec']['record_type'],
        'data_filter': json.dumps(definition['filter_spec'].get('data_filter', {})),
        'date_from':   user_filters['date_from'],
        'date_to':     user_filters['date_to'],
        'ps_id':       user_filters.get('ps_id')
    })

    # Add Sr. No.
    df.insert(0, 'Sr. No.', range(1, len(df) + 1))

    # Rename columns to label_en from field_registry
    rename_map = {k: fields_meta[k]['label_en'] for k in field_keys}
    df.rename(columns=rename_map, inplace=True)

    # Write to openpyxl sheet
    wb = openpyxl.Workbook()
    ws = wb.active
    ws.title = definition['header']['short_name_en']
    write_header_rows(ws, definition['header'], user_filters)
    write_dataframe(ws, df)
    return wb

Daily Diary — composite loop

def generate_composite(definition, user_filters, selected_ids=None):
    wb = openpyxl.Workbook()
    wb.remove(wb.active)   # remove default empty sheet

    sub_ids = selected_ids or definition['sub_template_ids']

    for sub_id in sub_ids:
        sub_template = load_template(sub_id)
        sub_def      = sub_template.template_definition
        sheet_name   = sub_def['header']['short_name_en']   # max 31 chars, pre-set in seed

        ws = wb.create_sheet(title=sheet_name)
        df = query_records(sub_def, user_filters)               # same function as single-sheet
        write_header_rows(ws, sub_def['header'], user_filters)
        write_dataframe(ws, df)

    return wb   # 34-sheet workbook written to one .xlsx file

main.py — RabbitMQ consumer

import pika, json, os
from dotenv import load_dotenv
from generator import generate_report

load_dotenv()

def on_message(channel, method, properties, body):
    payload = json.loads(body)
    job_id  = payload['job_id']
    try:
        generate_report(job_id)
        channel.basic_ack(delivery_tag=method.delivery_tag)
    except Exception as e:
        print(f"[ERROR] Job {job_id} failed: {e}")
        # Mark job as FAILED in DB, then ack so message doesn't loop
        mark_job_failed(job_id, str(e))
        channel.basic_ack(delivery_tag=method.delivery_tag)

def start():
    conn   = pika.BlockingConnection(pika.URLParameters(os.getenv('RABBITMQ_URL')))
    ch     = conn.channel()
    ch.exchange_declare(exchange='pharos', exchange_type='topic', durable=True)
    ch.queue_declare(queue='report-generation-queue', durable=True)
    ch.queue_bind(exchange='pharos', queue='report-generation-queue',
                  routing_key='report.requested')
    ch.basic_qos(prefetch_count=1)   # process one job at a time
    ch.basic_consume(queue='report-generation-queue', on_message_callback=on_message)
    print("[INFO] Python worker waiting for report jobs...")
    ch.start_consuming()

if __name__ == '__main__':
    start()

// Replace setImmediate(generateReportInternal) with:
import { publish } from '../../events/eventBus.js';

export const generateReport = async (req, res) => {
  const { template_id, custom_definition, filters, format, selected_sub_templates } = req.body;

  // Validate template access
  if (template_id) {
    const tpl = await db('report_templates').where({ id: template_id, is_active: true }).first();
    if (!tpl) return res.status(404).json({ success: false, message: 'Template not found' });
  }

  // Validate custom field_keys exist in field_registry
  if (!template_id && custom_definition) {
    await validateCustomDefinition(custom_definition);  // throws if invalid key
  }

  // RBAC scope: HC cannot generate reports for other PS
  if (req.user.role === 'HC' && filters.ps_id && filters.ps_id !== req.user.ps_id) {
    return res.status(403).json({ success: false, message: 'Cannot generate reports outside your PS' });
  }

  const jobId = uuidv4();
  await db('report_jobs').insert({
    id: jobId, template_id: template_id || null,
    custom_definition: custom_definition ? JSON.stringify(custom_definition) : null,
    filters: JSON.stringify(filters || {}),
    format: format.toUpperCase(), status: 'PENDING',
    file_path: path.join(reportsDir, `${jobId}.${ext}`),
    created_by: req.user.id, created_at: new Date(), updated_at: new Date()
  });

  // Hand off to Python worker via RabbitMQ
  await publish('report.requested', {
    job_id: jobId, template_id, custom_definition,
    filters, format: format.toUpperCase(),
    selected_sub_templates,
    user_id: req.user.id
  });

  return res.status(201).json({ success: true, data: { job_id: jobId, status: 'PENDING' } });
};

10 Field normalisation — the full prevention chain

SELECT fields must always store normalised values (e.g. E_BURGLARY not E-Burglary). If they don't, report filters silently return zero rows. This is prevented at every layer:

LayerWhereWhat it does
1. Import templatedownloadImportTemplate controllerExcel dropdown forces valid values. User cannot type freely in SELECT columns.
2. Import validationimport.service.js validateRow()Dry-run catches any value not in field_registry.options[]. Shown as INVALID_VALUE error before commit.
3. Normal form submitrecords.service.js createRecord()Server-side check: for each SELECT field, value must be in options[].value. Throws 422 if not. Frontend dropdown also enforces this, but server check is authoritative.
4. Empty report guardPython worker generator.pyIf query returns zero rows, writes a notice to the sheet explaining possible filter mismatch. No silent empty files.

Server-side validation to add in records.service.js

// In createRecord() and updateRecord() — add this before the DB insert
const selectFields = allFields.filter(f => f.field_type === 'SELECT');

for (const field of selectFields) {
  const submitted = data[field.field_key];
  if (!submitted) continue;   // required check is separate

  const options = parseJsonField(field.options);
  const valid   = options.map(o => o.value);

  if (!valid.includes(submitted)) {
    throw new ApiError(422,
      `Invalid value '${submitted}' for '${field.label_en}'. Valid: ${valid.join(', ')}`
    );
  }
}

Polling pattern (used for report jobs)

// hooks/useReportJob.js — create this file
import { useState, useEffect } from 'react';
import api from '../utils/api';

export function useReportJob() {
  const [jobId, setJobId]   = useState(null);
  const [status, setStatus] = useState(null);
  const [error, setError]   = useState(null);

  useEffect(() => {
    if (!jobId || status === 'READY' || status === 'FAILED') return;
    const interval = setInterval(async () => {
      const res = await api.get(`/reports/status/${jobId}`);
      const s   = res.data.data.status;
      setStatus(s);
      if (s === 'FAILED') setError(res.data.data.error_message);
    }, 2000);
    return () => clearInterval(interval);
  }, [jobId, status]);

  const download = () => {
    const token = localStorage.getItem('access_token');
    const link  = document.createElement('a');
    link.href     = `/api/v1/reports/download/${jobId}`;
    link.download = '';
    document.body.appendChild(link);
    link.click();
    document.body.removeChild(link);
  };

  return { setJobId, status, error, download, isReady: status === 'READY' };
}
