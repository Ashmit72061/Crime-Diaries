PHAROS System Documentation

Police Hierarchical Automated Reporting & Operations System — Complete Technical Reference

What is PHAROS?

PHAROS is a full-stack, multi-tenant police operations platform for managing FIR records, arrests, missing persons, UIDB entries, and daily diary reporting across an entire police district hierarchy — from head constable at the police station level up through HQ analysts.

Zero Hardcoded Forms

Every form field is a database row in field_registry. Adding a new field to any form never requires a code deploy.

Append-only Audit

Records are never deleted. Every mutation writes an immutable revision. Full history is available for any record at any time.

Hierarchy as Config

PS, District, JCP, SCP, HQ are rows in hierarchy_nodes. Adding a new station = one INSERT, zero code change.

Event-driven Modules

All cross-module communication goes through RabbitMQ. No module imports another module's service directly.

Bilingual

Every label carries label_en + label_hi. i18n is a first-class citizen in the DB schema, not an afterthought.

Async Report Engine

Excel / PDF generation is off-loaded to a Python worker via a job queue. The API responds instantly with a job ID; client polls for completion.

RBAC Implementation

Two middleware functions secure every route:

// 1. authMiddleware — verifies JWT, populates req.user
// 2. enforceScope  — sets req.jurisdictionQuery based on role

// Usage in router:
router.use(authMiddleware, enforceScope);
router.post('/', allow('HC'), controller.create);

// enforceScope sets req.jurisdictionQuery:
// HC / SHO           → { ps_id: user.ps_id }
// DISTRICT_OFFICER   → { district_id: user.district_id }
// JCP                → { sub_div_id: user.sub_div_id }
// HQ_ANALYST / HQ_ADMIN / SYSTEM_ADMIN → {} (global)

// In service — always filter with jurisdictionQuery:
if (jurisdictionQuery.ps_id)
  query = query.where('records.ps_id', jurisdictionQuery.ps_id);
if (jurisdictionQuery.district_id)
  query = query.where('records.district_id', jurisdictionQuery.district_id);

Per-record access uses verifyRecordAccess(recordId, user) — called on every single-record operation to confirm geographical ownership before allowing read or write.

Field Registry — The Form Engine

The field_registry table is the single source of truth for every form the system renders. The frontend fetches GET /api/fields/form/:record_type and renders whatever rows come back — no hardcoded field lists anywhere.

Supported Field Types

TEXT TEXTAREA NUMBER DATE TIME SELECT MULTISELECT BOOLEAN PHONE EMAIL FILE

Conditional Fields — show_when

Any field can be conditionally shown using the show_when JSONB column. Supports single value or array of values:

{
  "field": "local_head",
  "value": ["M.V. Theft", "Cycle Theft", "Snatching"]
}

Both DynamicForm families evaluate this client-side — the Ant Design form uses Form.useWatch() for reactive updates; the multi-step form evaluates it during section rendering and validation.

Record Types

CASE

FIR / case registration. Includes crime head, sections, complainant, accused, property, IO assignment. Drives 17 of the 34 daily diary sheets.

ARREST

Arrest record linked to a FIR or DD. Tracks arrested person, recovery, special scheme flags, juvenile data, financial fraud fields.

MISSING

Missing person report. Physical description, source (PCR/DD), tracing status. Drives sheets 18–23.

UIDB

Unidentified Dead Body. Physical description, inquest sections, found place/date. Sheet 19.

PCR_CALL

PCR call log entry. Operator, caller, action taken, time, beat assignment.

Adding a new field — the correct process INSERT a row into field_registry with the appropriate field_key, field_type, labels, and applicable_record_types. The form renders it automatically. Data is stored as a new key in the JSONB blob. No ALTER TABLE. No code change.

Every Transition Writes

  • A row in workflow_transitions (from/to status, action, comment, timestamp)
  • A row in record_revisions (change_type = 'STATUS_CHANGE')
  • A row in audit_logs
  • Publishes record.status_changed event to RabbitMQ

District Compilation

A compilation is a district-level roll-up of approved records for a given period, submitted to HQ. DISTRICT_OFFICERs create compilations; they transition from DRAFT → SUBMITTED.

GET /api/record-links/person-search?q=name — searches records.data JSONB across all record types for matching names, enabling officers to find existing records before creating a link.

Audit Trail

PHAROS maintains two audit streams, both append-only and never deleted:

record_revisions

One row per record mutation. Stores field_changes JSONB (diff of old→new data), change_type (CREATED / UPDATED / STATUS_CHANGE / HEAD_OVERRIDE), IP address, and optional reason text. Revision numbers are sequential per record.

audit_logs

Field-level log: one row per changed field. Stores old_value, new_value, changed_by_role, table name, IP address. Driven by the RabbitMQ record.* event subscriber in auditHandler.js.

Case Head Override PATCH /api/records/:id/override requires a reason string of at least 10 characters. Creates a HEAD_OVERRIDE revision entry with the original and new case head values.

Event Bus — RabbitMQ

All cross-module communication goes through a topic exchange named pharos. Modules publish events; handlers subscribe independently. No module imports another module's service file.

Python Worker

A standalone Python 3.11 process (python_worker/generator.py) consumes report.requested events from RabbitMQ and generates Excel / PDF files.

Worker Routing Logic

def generate_report(job):
    custom_definition = job.get('custom_definition', {})
    template = job.get('template_id')

    if job['format'] == 'EXCEL':

            wb = generate_daily_diary_workbook(custom_definition)

            wb = generate_linked_report(template, job['filters'])
        else:
            wb = generate_standard_excel(template, job['filters'])
        wb.save(job['file_path'])

    elif job['format'] == 'PDF':
        generate_pdf(template, job['filters'], job['file_path'])

Daily Diary Workbook (generate_daily_diary_workbook)

  • Receives pre-classified sheet data via custom_definition.sheets — zero DB queries in Python
  • Creates one Excel worksheet per REPORTS entry
  • Each sheet: title row → date row → spacer → styled header row → data rows with zebra striping
  • Header style: dark blue fill (#1F3864), white bold font, thin borders, text wrap
  • Data rows: alternating white / light blue (#DCE6F1)
  • Auto-fit column widths based on content

Conditional Fields — Client-side

The Ant Design DynamicForm wraps each field in a ConditionalFormItem component that calls Form.useWatch(field.show_when.field, form) for reactive visibility. The multi-step form evaluates show_when in FormSection.jsx during rendering and in the validation step before submit.

Both support array values in show_when.value:

// show_when evaluation (both families)
const allowed = Array.isArray(targetValue)
  ? targetValue.map(v => String(v).toLowerCase())
  : [String(targetValue || '').toLowerCase()];
if (!allowed.includes(current)) return null; // hide field

Migrations

Two migration systems co-exist:

1. Legacy script — backend/scripts/migrations.js

Raw SQL via Knex, with hasTable / hasColumn guards for idempotency. Run once on first deploy: node scripts/migrations.js. Creates all 12 core tables plus link tables.

2. Knex migrations — backend/migrations/

Standard Knex migration files with up() / down(). Run with npx knex migrate:latest.

Migration FileWhat It Does

Known Issues

1 — Orphaned *.routes.js files Old *.routes.js files exist in some modules. Only *.router.js files are imported by app.js. Do not import the old files.
2 — validate.middleware.js not wired The file exists but no route uses express-validator chains. API bodies are validated only with basic existence checks at the controller layer.
3 — Notification subscriptions notifications.service.js initSubscriptions() uses a different subscribe signature. Use notifyHandler.js init() instead, which is called correctly in app.js startServer().
4 — Report templates are in-memory Templates are hardcoded arrays in reports.controller.js, not DB rows. Adding a new standard template requires a code deploy.
5 — Dual router registration Both /api/ and /api/v1/ are registered for backward compatibility. All new code should use /api/.
